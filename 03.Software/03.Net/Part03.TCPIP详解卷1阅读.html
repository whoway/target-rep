      # 1.150行代码拉开协议栈实现的篇章

      对于网络的概念好像都懂，但是细问又不懂

      原因：

      1、大学的计算机网络，学习目的是考试。后边在学校也没写多少代码，关于网络相关的。

      2、工作的网络和书本上的不一样，大量以业务为主，而且不是TCP的网络，是http，这些基于tcp的

      3、百度查，为啥好多time_wait啥的

      # 关于网络的万金油的学习方法：

      1、站在==1个协议栈的设计的==角度，比如你花费1-2个月的时间，你尝试，自己造一个网络的框架，自己实现1个tcp的协议栈

      a.你对所有网络的原理，你都会懂，比如提到什么三次握手，什么epoll，你都会中间的原理和具体实现的流程

      学网络最好的方法不是看书+不是看技术博客，最好的是，你自己带着1个设计者的角度，你去实现1个TCP的协议栈。

      • 当你实现1遍之后，再来看书《TCP卷1》你就会发现你很多点的理解都会不一样。

      《TCP卷1》这样的书，是通过外文翻译过来的，翻译过程中，有些概念可能和你看到的技术博客上的，名词一样，但是表达的意思好像不一样==。那其实在看书的时候，在没有代码量的时候，我也不建议大家去啃书==，因为这个看到这个，你会越看越迷。

      如果看书看不懂，多来听听carl老师讲课、。。。听课比看书去更容易去吸收

      关于这个并发，很多朋友在聊。

      这个并发和框架没有半毛钱关系

      这个底层网络并发，IO连接的数量的管理，他和网络框架没有半毛钱关系，他是和epoll有关系

      就是你什么框架都 不用，就单跑1个epoll，不到100-200行代码，照样也能维护个1-2百万

      但是关于epoll的一些做法，以及关于他的一些优化的东西，一些各种场景上的问题就会比较多

      关于协程这个概念，其实80年代就有了，但是现在对于协程的概念是什么

      • 这个不比Http服务器更好？

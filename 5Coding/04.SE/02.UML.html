                  # 3.UML图中关系分类（4种和细分）

                  UML中的关系主要包括4种:

                  • 1）泛化关系（generalization）:代码层次实现：继承>
                  • 2）实现关系（realization）：代码层次实现：C++可以用纯虚函数和抽象类来模拟一个接口
                  • 3）依赖关系（dependency）:代码层次实现：“一个类是另一个类”的函数参数或函数返回值实现
                  • 包含关系（<<include>>）
                  • 扩展关系（<<extend>>）
                  • 4）关联关系（association）:代码层面是用“一个类是另外一个类”的成员变量
                  • 一般关联
                  • 聚合
                  • 组合
                  • 注意：关系逐渐加强。一般关联-->聚合-->组合

                  # （1）泛化关系

                  • 意义：在 UML 中, 泛化关系用来表示类与类, 接口与接口之间的继承关系. 泛化关系有时也称为”is a kind of”关系
                  • 图形：在 UML 中泛化关系用一条实线，空心箭头从子类指向父类

                  # （2）实现关系

                  • 意义：在 UML 中, 实现关系用来表示类与接口之间的实现关系.
                  • 图形：在 UML 中实现关系用一条虚线，空心箭头由子类指向父类

                  PS:实现关系，在C++语言里面没有接口这个概念，Java中有。 但是C++可以用纯虚函数和抽象类来模拟一个接口。 纯虚函数的类是抽象类。可以用抽象类弄成接口类，就这么干。

                  # （3）依赖关系

                  对于两个相对独立的系统，当一个系统负责构造另一个系统的实例，或者依赖另一个系统的服务时，这两个系统之间体现为依赖关系. 例如生产零件的机器和零件，机器负责构造零件对象;充电电池和充电器，充电电池通过充电器来充电;自行车Bicycle和打气筒Pump，自行车通过打气筒来充气

                  # 依赖关系之包含

                  • 一个用例可以包含其他用例具有的行为, 并把它包含的用例行为作为自身行为的一部分.
                  • 图形：在 UML 中包含关系用虚线箭头加 “<<include>>”,箭头指向被包含的用例

                  # 依赖关系之扩展

                  • 如果在完成某个功能的时候偶尔会执行另外一个功能, 则用扩展关系表示.
                  • 图形：在 UML 中扩展关系用虚线箭头加 “<<extend>>”, 箭头指向被扩展的用例

                  说明：在保存成绩的时候，只有发现学生考试没有及格，才需要扩展到“打印补考通知单”。（偶尔，毕竟并不总是需要打印补考通知单）

                  # （4）关联关系

                  关于依赖是虚线，关联是实现的对比

                  张三这次开车上班，车是自己的。

                  • 在 Java和C++ 中，关联关系是通过使用成员变量来实现的。 如人与车

                  # 2、聚合（空心菱形+实线）

                  也是属于关联，只不过是车和人之间的关系会更加紧密一些而已，

                  • 聚合是整体和部分之间的关系，例如汽车由引擎、轮胎以及其它零件组成。

                  聚合关系也是通过成员变量来实现的。

                  对比： 关联关系所涉及的两个类处在同一个层次上 而聚合关系中，两个类处于不同的层次上，一个代表整体，一个代表部分。

                  # 3、组合（实心菱形+实线）

                  聚合和组合的区别： 有无生命的区别。生命中的整体和部分 比如,公司和部门，公司不存在了，则部门也没有了。要是部门不存在类，公司也没有存在的意义。

                  • 代表整体的对象负责代表部分对象的生命周期（重点理解！！！）

                  # 关联关系的多重性（软考考）

                  关联还有一个数量问题

                  • 关联关系的多重性是指有多少对象可以参与该关联, 多重性可以用来表达一个取值范围, 特定值, 无限定的范围.

                  • 在UML中，多重性可以用下面的格式表示： 0..1 0..（也可以表示为0..n） 1（1..1的简写） 1..（也可以表示为1..n） *（即0..n） 7 3,6..9 0（0..0的简写）（表示没有实例参与关联，一般不用） 可以看到，多重性是用非负整数的一个子集来表示的。

                  比如这样表示，一个人可以有1,2,3...n辆车

                  举例： 前面聚合的例子： 一个车站可以有1个引擎，4个轮胎

                  # 4.1.“聚合”和“组合”的区别是什么？

                  组合和聚合是有很大区别的，这个区别不是在形式上，而是在本质上：

                  比如zhiA类中包含B类的一个dao引用b，当A类的一个对象消亡时，b这个引用所指向的对象也同时消亡（没有任何一个引用指向它，成了垃圾对象），这种情况叫做组合，反之b所指向的对象还会有另外的引用指向它，这种情况叫聚合。

                  在实际写代码时组合方式一般会这样写： A类的构造方法里创建B类的对象，也就是说，当A类的一个对象产生时，B类的对象随之产生，当A类的这个对象消亡时，它所包含的B类的对象也随之消亡。 聚合方式则是这样： A类的对象在创建时不会立即创建B类的对象，而是等待一个外界的对象传给它 传给它的这个对象不是A类创建的。

                  现实生活中： 人和人和手，脚是组合关系，因为当人死亡后人的手也就不复存在了。人和他的电脑是聚合关系。 class Hand{ } class Computer{ }

                  组合： class Person{ private Hand hand; public Person(){ hand = new Hand(); } }

                  聚合： class Person{ private Computer computer; public setComputer(){ computer = new Computer(); } }

                  PS:个人觉得人和手那样设计不好 问：人和他的电脑不是依赖关系吗？ 答：要看你是从那个角度设计的吧，都有可能

                  # ✅静态模型图

                  # 1.用例图（★★★）

                  用例图： 画出来的，要是画出来之后给别人看，没有很大的歧义就算是可以了。

                  没有板上钉钉的事情，没有错误之分，只是更加合适一点。

                  # 2.类图（★★★★★）

                  类是类图的主要组件, 由 3 部分组成: 类名, 属性和方法. 在 UML 中, 类用矩形来表示,

                  • 顶端部分存放类的名称
                  • 中间部分存放类的属性, 属性的类型及值
                  • 底部部分存放类的方法, 方法的参数和返回类型

                  在 UML 中可以根据实际情况有选择的隐藏属性部分或方法部分或两者都隐藏

                  • 在 UML 中, 公用public类型有 + 表示, 私有private类型用 – 表示, protect保护类型用 # 表示.
                  • UML 的工具开发商可以使用自己定义的符号表示不同的可见性

                  # ✅动态模型图

                  # 1.用例图（★★★）

                  # 1、用途：

                  用例图(Use Case Diagram): 也称为用户模型图, 是从软件需求分析到最终实现的第一步, 它是从客户的角度来描述系统功能。 所以是 1）站在客户如何使用系统的角度，把整个软件系统描述出来。 2）就是把客户的需求给固定下来。 3）画的图能进行交流就好了，仁者见仁智者见智的过程，并不要求每个人画的用例图完全一样。

                  # 2、组件

                  参与者(Actor) 用例(Use Case): 代码层面，用例，你就把它想象成是一个简单的功能模块就好了。 关系

                  # 3、用例之间关系

                  • 1）泛化：代码层次是用“继承”（C++或Java）
                  • 2）依赖：代码层面是用，“一个类是另一个类”的函数参数或函数返回值实现
                  • 包含:一个用例可以包含其他用例具有的行为，也就是说，这个功能模块要使用另一个功能模块。 eg.教学管理系统，添加学生信息（用例）要使用更新数据库（用例）这个功能模块。
                  • 扩展：完成某个功能的时候偶尔会执行另外一个功能, 则用扩展关系表示。 eg.比如，在保存成绩的时候，发现，这个学生没有考试及格，以前的做法，就需要补考，就需要扩展到打印补考通知单。（偶尔）

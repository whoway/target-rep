        # 1.顶层引导概述「Toplevel-bootstrap overview」

        顶级引导程序能够在所有三个阶段通过单独的重新配置和重新配置 libiberty/libcpp/gcc来引导编译器。实际上可以引导整个工具链，以便最终的可执行文件完全使用组合树「combined tree」中的汇编器、binutils 和链接器构建。

        在本地环境中进行配置时，make或多或少地做了一些make bubblestrap过去所做的事情：从 stage1 开始，重新构建必须重新构建的所有内容，如果尚未配置 stage2，则配置 stage2，构建 stage2，对于 stage3 也是如此。“重建”不限于GCC：libiberty、libcpp等GCC的依赖都是经过3次配置编译。这与从头开始重建整棵树三次相同，每次都使用以前的构建作为结果。

        它支持所有的花里胡哨。 make profiledbootstrap工作；当在组合树中运行时，它将构建配置文件优化的汇编器、binutils 和链接器。

        # 2.顶层引导-备忘录

        下面给出了常用操作模式的备忘单。

        Q1) 如何引导（bootstrap）编译器去包含所有库（including all libraries）？

        A1）从顶层目录make

        Q2) 如何引导编译器，但不构建任何耗时的库（libjava、libstdc++）？

        A2)make stage{2,3,feedback}-bubble

        注意：这分别相当于旧的cd gcc && make {bootstrap2,bootstrap,profiledbootstrap} 。

        Q3) 如何只编译 stage1 编译器 + 基本运行时（libgcc、libmudflap、libgomp）？

        A3)make stage1-bubble all-target-libmudflap all-target-libgomp

        注意：我也在考虑三阶段 libmudflap 和 libgomp；在这种情况下，以上将简化为make stage1-bubble

        Q4) 如何使用 stage1 编译器编译所有库（libjava，...）？

        A4) 使用**--enable-stage1-languages=all进行配置，然后使用make stage1-bubble all-target**

        注意：我正在考虑在非发布分支中默认启用此标志。当然，对于 GCC 开发人员来说，总是用这个标志配置他们的树是最有用的。

        Q5）如何清理所有东西（包括stage1编译器，但不包括makefile，这样我可以从头开始构建所有东西，但不必运行configure）？

        A5) make clean（或 make cleanstrap 也启动一个引导程序）

        Q6) 如何清理除 stage1 编译器之外的所有内容？

        A6) make distclean-stage2 clean-target或make restrap（这也将启动引导程序）
